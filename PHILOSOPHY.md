# CodeScaffold 设计哲学

## 核心原则

### 1. AI 是可选的，不是必须的

CodeScaffold 的核心价值在于**组件化架构**，AI 只是增强功能。

- 离线也能用（纯模板模式）
- AI 是增强功能（需要 API key 才激活）
- 速度优先（< 3 秒初始化）

### 2. 组件化优于模板化

不是提供固定模板，而是让用户**自由组合**组件：

- 框架 + 数据库 + 认证 + UI = 完整项目
- 每个组件独立、可替换
- 组件之间有依赖关系和兼容性检查

### 3. 安全第一

- 路径遍历验证（支持 Windows 和 Unix）
- GitHub 模板源格式验证
- 组件兼容性检查
- 代码质量门禁

### 4. 开发者体验优先

- 零配置启动（`--defaults`）
- 偏好记忆（自动保存上次选择）
- 实时预览（选择组件时显示项目结构）
- 智能推荐（根据框架推荐最佳组件）

### 5. 不膨胀原则

- 只包含必要的依赖
- 按需添加组件
- 最小化模式（`--minimal`）
- 空模式（`--empty`）

## 设计决策

### 为什么选择组件化而不是模板化？

模板化是"给你一个完整的项目"，组件化是"让你选择需要的部分"。

**模板化的问题：**
- 模板可能包含你不需要的东西
- 难以定制
- 维护成本高

**组件化的优势：**
- 只包含你需要的
- 完全可定制
- 易于维护和扩展

### 为什么 AI 是可选的？

AI 依赖外部 API，可能：
- 网络不可用
- API key 未配置
- 响应时间不稳定

所以 AI 必须是可选的增强功能，不是核心依赖。

### 为什么支持离线模式？

开发者可能：
- 在没有网络的环境工作
- 使用公司内网
- 需要快速启动

离线模式确保 CodeScaffold 在任何环境下都能工作。
